# Eliminate Code Duplication in venom-router-react

**Date:** 2026-06-23
**Status:** Approved — pending implementation
**Approach:** Option A — Delete the dead `createServerFn` files, keep the live HTTP dispatcher

## Context

The project ships **two parallel implementations** of the same dashboard business logic:

1. **Live HTTP dispatcher** at `src/lib/api/dashboard-router.server.ts` (1031 lines) — the only path the browser actually uses. Wired through `src/lib/api-client.ts` → `/api/dashboard/*` → `handleDashboardAPI`.
2. **Dead `createServerFn` wrappers** in `src/lib/providers/integrations.functions.ts` (14 wrappers) and `src/lib/venom.functions.ts` (17 wrappers) — never imported by any client or server code. Verified by project-wide grep.

The "duplication" the owner perceived is actually duplication with dead code. The dead code happens to be auto-generated by a TanStack Start + Lovable Cloud scaffold. The owner has **stopped using Lovable** and wants the Lovable-specific plumbing removed.

**This refactor deletes dead code, eliminates the duplication that comes with it, and removes Lovable's auto-generated authentication middleware.** Zero behavior change for the live UI.

- **Keep** the internal helpers that ARE consumed by `dashboard-router.server.ts`:
  - `syncAccountInternal`
  - `toSyncAccountWireResponse`
  - `aggregateCatalogModels`
  - `runAntigravityLiveSnapshotFetch`
  - `runProviderModelSync`
  - `extractCapabilityList`
- **Keep** the exported `CatalogModel` type
- Update all imports in `dashboard-router.server.ts` to point at the new path
- The `.functions.ts` suffix was a TanStack Start convention for `createServerFn` wrappers; once the wrappers are gone, the suffix is misleading, hence the rename to `.service.ts`

#### `src/start.ts` (lines 6–19)

- Remove `functionMiddleware: [attachSupabaseAuth]` registration (the attacher is deleted)

## Architecture after refactor

```
Browser
  └─ src/lib/api-client.ts (authFetch — injects Supabase session Bearer)
       └─ fetch("/api/dashboard/{resource}/{id?}/{sub?}")
            └─ src/server.ts:107 (worker fetch)
                 └─ handleDashboardAPI(request)
                      ├─ requireDashboardAuth(request)         [src/lib/dashboard-auth.server.ts]
                      │      └─ supabaseAdmin.auth.getClaims(token)
                      ├─ parseBody / schema.parse              [zod schemas inline]
                      └─ dispatch by path → handler
                           ├─ supabase.* / adapter.*           [src/lib/providers/adapters/*]
                           └─ syncAccountInternal, etc.        [src/lib/providers/integrations.service.ts]
```

**No second transport. No Lovable plumbing. No `createServerFn` stubs. One HTTP API surface.**

## Testing

### Problem

`dashboard-router.server.ts`, `chat-completions.server.ts`, `integrations.functions.ts`, `venom.functions.ts`, and `auth-middleware.ts` currently have **zero direct test coverage**. The refactor removes dead code, but without a test layer we have no regression net.

### Plan

Add a new integration test file at `src/lib/api/dashboard-router.test.ts` that:

- Mocks the `supabaseAdmin` client (returns canned query results)
- Mocks the dynamic adapter imports (`@/lib/providers/adapters/*`)
- Calls `handleDashboardAPI` directly with a constructed `Request` object
- Asserts on the response shape and status code

**Minimum test cases** (covers the critical paths without inflating scope):

| Test | Endpoint | Expected |
|------|----------|----------|
| Owner auth — missing header | any | 401 |
| Owner auth — non-owner JWT | any | 401 |
| `GET /api/dashboard/metrics` | metrics | 200 + correct shape |
| `POST /api/dashboard/oauth/start` | oauth flow start | 200 + `{flow_id, authorize_url}` |
| `POST /api/dashboard/api-keys` | key creation | 200 + `{id, raw, prefix}` |
| `DELETE /api/dashboard/accounts/:id` | disconnect | 200 |
| Zod validation failure | any POST | 400 + error message |

Existing tests (10 files under `src/lib/db/*.test.ts` and `src/lib/providers/*.test.ts`) must continue to pass unchanged.

## Risks

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| `start.ts` SSR code depends on `attachSupabaseAuth` | Low | Search the project for any reference to `attachSupabaseAuth` before deletion; remove the `functionMiddleware` line only after confirming zero references |
| `client.server.ts` marked "auto-generated" may be regenerated by some external process | Low | Owner confirmed Lovable is no longer used; even if a future process regenerates it, the new content must produce the same supabase admin client interface we depend on |

## Out of scope (intentional)

- **The 32 dynamic `await import(...)` calls** in `dashboard-router.server.ts` — pre-existing pattern, not blocking, and unrelated to duplication
- **`chat-completions.server.ts`** — has its own auth (API key, not owner JWT) and its own OpenAI-compatible error envelope; not duplicated with anything
- **Adding tests to `chat-completions.server.ts`** — separate surface, separate auth model, no duplication driver
- **Adopting `createServerFn` again** — the project has moved past it; the owner is removing TanStack-specific plumbing
